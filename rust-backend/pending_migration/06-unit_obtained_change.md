# unit_obtained_change — Rust 500: sqlx decode type mismatch on unit_time

## Summary

The websocket-sync key `unit_obtained_change` returns **HTTP 500** from the Rust
backend while Java returns **HTTP 200** with data on the same `owge` DB.

The Rust finder is `ObtainedUnitBo::find_completed_dtos`
(`rust-backend/owge-business/src/bo/obtained_unit_bo.rs:345`), reachable from the
dispatch arm in `rust-backend/owge-business/src/websocket/sync.rs:69-71`.

The failure is a `sqlx` row-decode error: a `DerivePartialModel` field declares the
`units.time` column as `Option<u64>` (which sqlx maps to `BIGINT UNSIGNED`), but the
actual `units.time` column is signed `INT`. sqlx refuses the mismatched decode and the
request fails before any data is returned.

The "u64 vs INT" diagnosis is **CONFIRMED**, with one refinement: the offending
`unit_time` column is **not** a hand-written SQL alias on this path. It is generated by
SeaORM's `DerivePartialModel`, which prefixes the nested `unit` model's columns with the
relation name, producing the column alias `unit_time` for `units.time`. (The hand-written
`u.time AS unit_time` aliases at `attack_mission_manager_bo.rs:1587` / `:1614` are on the
**combat/attack** path, are decoded by a different struct that already uses the correct
signed type, and are not involved in this 500.)

## Observed error

Rust server log on `GET /game/websocket-sync?keys=unit_obtained_change`:

```
ERROR request failed error=Query Error: error occurred while decoding column "unit_time":
mismatched types; Rust type `core::option::Option<u64>` (as SQL type `BIGINT UNSIGNED`)
is not compatible with SQL type `INT`
```

Java returns **HTTP 200** with the obtained-unit list for the same user on the same DB,
so this is a Rust-only decode regression, not a data problem.

## Root cause

The single joined query built by `ObtainedUnitBo::with_full_joins`
(`obtained_unit_bo.rs:465`) is decoded into the partial model `ObtainedUnitJoinedRow`
(`obtained_unit_bo.rs:150-166`). Its `unit` field is `#[sea_orm(nested)]`
(`obtained_unit_bo.rs:156-157`) of type `UnitJoinedRow`. Because `UnitJoinedRow` is a
partial model over a **different** entity (`unit::Entity`), SeaORM's `DerivePartialModel`
aliases each of its columns with the `unit_`-prefixed form — so `units.time` is selected
`AS unit_time`. That is the source of the literal `unit_time` column in the error (no
hand-written SQL alias exists on this path).

The offending field:

- **`rust-backend/owge-business/src/bo/obtained_unit_bo.rs:50`** —
  in `struct UnitJoinedRow`:
  ```rust
  time: Option<u64>,
  ```
  This decodes `units.time` as `BIGINT UNSIGNED`.

Verified column type:

- Schema `business/database/02_schema.sql:1027`:
  ```sql
  `time` int DEFAULT NULL COMMENT 'El tiempo base para la mejora, en segundos',
  ```
- Live DB confirms it (`information_schema.COLUMNS`): `units.time` COLUMN_TYPE = **`int`**
  (signed, nullable). No migration under `business/database/migrations/` alters it.
- The SeaORM entity itself already declares it correctly:
  `rust-backend/owge-business/src/bo/obtained_unit_entity/unit.rs` →
  `pub time: Option<i32>,`. The hand-written partial model `UnitJoinedRow` diverged from
  the entity.

So `units.time` is signed `INT` → must decode to `Option<i32>` (or `Option<i64>`), not
`Option<u64>`. That single field mismatch is the 500.

### Additional latent mismatches in the same struct (will surface after the first fix)

sqlx reports only the **first** column that fails to decode. Two more fields in
`UnitJoinedRow` carry the same class of signed/unsigned-width mismatch and would throw the
identical error once `time` is corrected:

- `obtained_unit_bo.rs:48` — `primary_resource: Option<u64>` — but `units.primary_resource`
  is `INT UNSIGNED` (sqlx → `u32`). Should be `Option<u32>`.
- `obtained_unit_bo.rs:49` — `secondary_resource: Option<u64>` — but
  `units.secondary_resource` is `INT UNSIGNED` (sqlx → `u32`). Should be `Option<u32>`.

(The entity `unit.rs` again declares both correctly as `Option<u32>`.) Every other
`UnitJoinedRow` field matches its column: `id: u16`←`smallint unsigned`,
`attack/health/shield/charge: Option<u16>`←`smallint unsigned`, `speed: Option<f64>`←
`double`, `stored_weight: u32`←`int unsigned`, `storage_capacity: Option<u32>`←
`int unsigned`, the `tinyint` flags as `u8`/`i8`. So the three resource/time fields are the
only offenders, all in the same struct.

## Proposed fix (NOT implemented)

**Recommended: fix the decode types at the partial-model layer** to match the actual
columns, and convert at the DTO boundary. In `UnitJoinedRow`
(`obtained_unit_bo.rs:48-50`):

```rust
primary_resource: Option<u32>,   // units.primary_resource  INT UNSIGNED
secondary_resource: Option<u32>, // units.secondary_resource INT UNSIGNED
time: Option<i32>,               // units.time INT (signed, nullable)
```

Then in `unit_dto_from_joined_row` (`obtained_unit_bo.rs:247-249`) convert to the
`UnitDto` types. The minimal, behaviour-preserving option keeps `UnitDto`'s public JSON
shape unchanged via widening casts:

```rust
time: r.time.map(|v| v as u64),               // or change UnitDto.time -> Option<i32>
primary_resource: r.primary_resource.map(|v| v as u64),
secondary_resource: r.secondary_resource.map(|v| v as u64),
```

Cleaner alternative (preferred if JSON parity is verified): change
`UnitDto.time`/`primary_resource`/`secondary_resource` (`dto/unit.rs:24-26`) to
`Option<i32>` / `Option<u32>` to mirror the column types and the already-correct
`ObtainedUnitUnitDto`, dropping the casts entirely. This must be checked against every
other `UnitDto` producer/consumer (e.g. `unit_bo::UnitRow`) so no other path breaks — do a
repo-wide build after the change.

Avoid the `CAST(... AS SIGNED)`-in-SQL route here: the alias is generated by SeaORM, not
hand-written, so there is no literal SQL string to edit without restructuring the partial
model — fixing the Rust field types is both simpler and matches the entity definition.

Fix all three fields in one change; fixing only `time` will move the 500 onto
`primary_resource` next.

## Risk & notes

- **Audit other signed/unsigned decodes.** This is the same bug class flagged elsewhere in
  the port: a `u64`/`u32`/`u16` partial-model or `FromRow` field over a signed/narrower
  column. After this fix, grep for other `AS unit_time` selects and other `UnitDto`
  builders to confirm they decode `time` signed:
  - `rust-backend/owge-business/src/bo/attack_mission_manager_bo.rs:1587,1614`
    (`u.time AS unit_time`) is decoded by a struct whose field is already
    `unit_time: Option<i32>` (`attack_mission_manager_bo.rs:86`) — **correct**, leave as is.
  - Sweep for `time: Option<u64>` / `primary_resource: Option<u64>` /
    `secondary_resource: Option<u64>` in any other partial model or `FromRow` reading the
    `units` table.
- The SeaORM entity (`obtained_unit_entity/unit.rs`) is the source of truth for column
  types and already correct; prefer aligning the hand-written partial model to it.
- Changing `UnitDto`'s field types (the cleaner alternative) has wider blast radius than
  the partial-model-only fix — verify JSON parity with Java (numbers serialize the same
  either way, but the field signedness must still represent the same range) and a full
  `cargo build`/test before choosing it.
- `units.time` is nullable, so the `Option` wrapper is required regardless of signedness.
